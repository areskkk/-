# 资格预判规则引擎设计

## 1. 定位

资格预判采用规则优先，AI 只做解释、总结和追问建议。AI 不得绕过结构化规则给出相反结论。

规则表达式统一采用 `docs/规则表达式DSL语法定义.md`。规则引擎只执行已发布政策版本中的已审核 DSL；AI 只能基于政策原文生成 DSL 草稿，必须经政策管理员审核发布后才能进入规则库。

## 2. 输入

| 输入 | 来源 |
| --- | --- |
| 企业画像快照 | 企业手填/后台导入 |
| 政策结构化条件 | PolicyCondition |
| 材料 OCR 字段 | Material OCR |
| 本次申报表单 | Application |

## 3. 输出

| 输出 | 含义 |
| --- | --- |
| eligible | 符合 |
| ineligible | 不符合 |
| need_info | 待补充 |
| manual_review | 需人工复核 |

## 4. 判定逻辑

| 场景 | 输出 |
| --- | --- |
| 所有必填条件满足 | eligible |
| 任一硬性条件明确不满足 | ineligible |
| 缺少必填字段或证据 | need_info |
| 条件冲突、政策语义不清、自由裁量 | manual_review |
| AI 解释与规则冲突 | manual_review，且以规则结果为准 |

### 4.1 DSL 执行边界

- `field_key/operator/target_value/required/evidence_type/fail_action` 为最小条件单元。
- `AND` 组合用于同一政策下多个硬性条件同时满足。
- `OR` 组合用于多个可替代证据或可选路径。
- 嵌套条件必须限制在政策结构化 Schema 内，不允许执行任意代码。
- `fail_action=ineligible` 的硬性条件明确失败时，规则结果优先于 AI 解释。
- `fail_action=need_info` 表示缺少画像字段或材料证据，企业端提示补充。
- `fail_action=manual_review` 表示自由裁量、语义不清、证据冲突或低置信度，进入人工兜底。

## 5. 排序策略

政策推荐排序只用于展示，不替代资格判断。

优先级：

1. 硬性条件满足度。
2. 必填材料完整度。
3. 政策有效期和申报截止时间。
4. 企业画像相关度。
5. 政策运营优先级。

## 6. 边界条件

| 边界 | 处理 |
| --- | --- |
| 政策已废止 | 禁止新申报，存量申报转人工 |
| 企业画像变更 | 草稿重新预判，已提交用快照 |
| 多政策申报 | 每个政策子项独立预判 |
| 材料 OCR 低置信度 | 不作为硬证据，需人工确认 |

## 7. AI 角色

- 把规则命中结果转成用户可读说明。
- 生成缺失字段追问。
- 汇总“不符合”的原因。
- 不得给出最终审批结论。
