# PRD：南康助企宝

> 产品定位：面向南康家具产业集群的惠企政策智能服务平台  
> 文档状态：Draft  
> 适用对象：产品、政务业务、数据、算法、研发、运维、审批部门  
> 本版依据：`PRD改进.md` 与 prd-planner 访谈确认结果  
> 主口径说明：本文为四份 PRD 的主口径，企业端、政府端、管理后台需与本文功能编号、范围和边界一致。

## 1. 背景

南康区家具产业企业数量多、惠企政策多、申报材料复杂。企业在政策咨询、资格判断、材料准备、申报跟踪中存在信息差和重复劳动；政府窗口和审批人员在政策解释、材料核验、条件比对、补正沟通中存在效率瓶颈。

“南康助企宝”建设“企业端 + 政府端 + 管理后台 + RAG/Agent 能力中台”，把政策查询、资格预判、申报辅助、材料核验、辅助审核、运营看板串成闭环。

## 2. 目标用户

| 用户类型 | 典型角色 | 核心诉求 |
| --- | --- | --- |
| 家具企业经营者 | 企业主、厂长、财务负责人 | 快速知道能申报什么、是否符合、怎么申报 |
| 企业办事人员 | 行政、财务、人事、项目申报人员 | 少填少传材料，清楚看到进度和补正要求 |
| 政府窗口人员 | 惠企窗口、园区服务人员 | 快速承接咨询，减少重复解释和人工分流 |
| 审批人员 | 工信、商务、人社、税务等部门审核人员 | 快速比对条件、核验材料、生成可追溯意见 |
| 管理部门 | 区级产业、政务服务、数字化主管部门 | 掌握政策兑现效率、企业覆盖率和办理堵点 |

## 3. 问题定义

### 3.1 企业侧问题

1. 政策分散，企业不知道哪些政策与自身相关。
2. 条款专业，企业难以判断是否满足条件。
3. 材料重复，尤其多政策申报时重复上传、重复填写。
4. 进度不透明，补正要求和办理时限不清楚。

### 3.2 政府侧问题

1. 重复咨询多，窗口解释成本高。
2. 条件比对依赖人工，初审效率低。
3. 部门权限、材料使用、数据同步边界不清。
4. 政策效果缺少结构化数据和可量化指标。

## 4. 目标与非目标

### 4.1 产品目标

| 目标 | 量化口径 |
| --- | --- |
| 降低政策获取成本 | MVP 政策问答有效响应率 >= 80%，答案引用率 >= 95% |
| 提升资格预判可用性 | 高频政策预判准确率 POC 后校准，初始目标 >= 85% |
| 降低本次申报材料重复 | 多政策申报中同类材料在本次申报内复用，减少重复上传 |
| 提升审核效率 | 二期平均审核耗时较基线缩短 >= 30% |
| 建立闭环运营 | 管理后台可追踪咨询、申报、审核、补正、AI 采纳和异常指标 |

### 4.2 非目标

- 不替代最终行政审批决定。
- 不在低置信度、证据不足、政策冲突场景下给出确定性审批结论。
- MVP 不对接“亲清赣商”“惠企通”，只做独立系统能力；外部平台对接作为后续扩展。
- MVP 不建设完整数据中台和模型训练平台。
- MVP 不做企业材料长期保存和跨事项长期复用，只支持本次申报内复用。

## 5. 已确认产品口径

| 决策项 | 已确认口径 |
| --- | --- |
| 手机端范围 | MVP 采用 H5 响应式，支持手机端完整申报、材料上传和多政策填报 |
| 外部平台 | 本期不对接亲清赣商/惠企通，只做独立系统能力 |
| 登录认证 | MVP 自建账号体系；企业绑定由 Agent 对低风险申请自动通过并留痕，人工后期可修改；预留赣服通、政务统一认证、政务 OA 对接 |
| 企业画像 | MVP 企业手填为主，后台可导入，未来再接权威数据源 |
| 多政策申报 | 前端合并填写，后端按政策拆成多条申报事项 |
| 资格预判 | 规则优先，AI 只做解释和辅助总结 |
| 材料复用 | 只允许本次申报内复用，不做长期材料库复用 |
| AI 审批意见 | MVP 允许生成草稿，但必须人工确认后生效 |
| 低置信度 | 写初始阈值，后续通过 POC 校准；人工兜底归平台运营队列，普通任务 1 个工作日内处理 |
| MVP 试点 | 按核心部门约 60 份政策、100 家企业规划 |

## 7. 用户流程

### 7.1 企业政策咨询

1. 企业输入文字或语音问题。
2. 系统识别意图和缺失画像字段。
3. 系统检索政策库，按权限和企业画像筛选。
4. 系统输出政策解释、来源引用、资格预判和下一步操作。
5. 信息不足时追问关键字段，不能判断时进入人工兜底。

### 7.2 企业申报

1. 企业查看推荐政策和符合度。
2. 选择一个或多个政策发起申报。
3. 系统预填企业画像；多政策共用材料在本次申报内复用。
4. 企业补充差异化字段和缺失材料。
5. OCR 和规则检查材料完整性、有效期、字段一致性。
6. 企业在桌面端或手机端确认提交。
7. 后端按政策拆分为多个申报事项，进入政府端预审和审核。

### 7.3 政府辅助审核

1. 审批人员进入任务队列。
2. 系统加载企业画像快照、政策条件、材料和历史记录。
3. 审核 Agent 执行条件比对和材料校验。
4. 系统输出预审建议、依据引用、风险点和审批意见草稿。
5. 审批人员确认、修改、补正、驳回或通过。
6. 人工结果回流标注库和运营指标。

## 10. 状态机

| 状态 | 触发来源 | 可流转到 | 说明 |
| --- | --- | --- | --- |
| draft | 企业新建 | submitted、cancelled | 草稿未提交，可编辑 |
| submitted | 企业提交 | pre_reviewing、withdrawn | 已提交，生成画像快照和材料版本 |
| pre_reviewing | 系统预审 | reviewing、need_supplement、manual_review | AI 和规则预审 |
| reviewing | 审批人员审核 | approved、rejected、need_supplement、manual_review | 人工审核中 |
| need_supplement | 审批人员发起补正 | resubmitted、withdrawn、timeout_closed | 企业需补充材料 |
| resubmitted | 企业补正提交 | pre_reviewing、reviewing | 补正后再次审核 |
| manual_review | 系统或人工标记 | reviewing、need_supplement、approved、rejected | 复杂事项人工复核 |
| approved | 审批通过 | archived | 办结通过 |
| rejected | 审批不通过 | archived | 办结不通过 |
| withdrawn | 企业撤回 | archived | 撤回归档 |
| timeout_closed | 超时未补正 | archived | 按规则关闭 |
| archived | 系统归档 | - | 只读 |

状态约束：

- `submitted` 后企业不得直接修改主申报信息，只能通过补正流程提交新版本。
- `need_supplement` 必须包含补正原因、材料清单、截止时间。
- 政策废止后，未提交草稿提示不可申报；已提交事项进入人工复核。
- 并发审核采用乐观锁，版本不一致时提示刷新。

## 11. 资格预判、低置信度与材料规则

### 11.1 资格预判

| 输出 | 判定规则 |
| --- | --- |
| 符合 | 所有必填规则条件均满足，材料要求已满足或可补齐 |
| 不符合 | 任一硬性必填条件明确不满足 |
| 待补充 | 缺少关键画像字段或证明材料 |
| 需人工复核 | 政策条件冲突、自由裁量、低置信度、证据不一致、权限不足 |

资格预判采用规则优先。AI 只能解释规则命中结果、总结差距和生成用户可读说明，不能替代规则判断。

### 11.2 低置信度初始阈值

| 阈值 | 初始值 | 处理 |
| --- | --- | --- |
| 检索 Top1 相似度 | < 0.75 | 不直接给确定答案，扩大检索或转人工 |
| OCR 字段置信度 | < 0.85 | 企业或审批人员人工确认 |
| 资格预判证据完整度 | 必填条件证据缺失 | 输出待补充 |
| 模型输出 | 无引用、与规则冲突、含明显不确定表述 | 转人工复核 |

以上为初始值，必须通过 POC 和真实样本校准。

### 11.3 材料复用

| 规则 | 要求 |
| --- | --- |
| 复用范围 | 仅限同一次申报内，多政策共用材料可复用 |
| 同文件识别 | 文件 hash 一致时可复用 |
| 有效期校验 | 按政策材料要求校验 issue_date、expire_date、validity_days |
| 版本留痕 | 申报引用材料时固化 material_id、file_id、hash 和引用时间 |
| 人工确认 | OCR 低置信度、材料类型不确定、字段不一致时需人工确认 |
| 长期保存 | MVP 不做企业材料长期保存和跨事项长期复用 |

## 12. 接口、认证与数据同步

### 12.1 外部系统

| 系统 | 本期范围 | 后续扩展 |
| --- | --- | --- |
| 亲清赣商/惠企通 | 本期不对接，不作为 MVP 依赖 | 后续视接口开放情况对接事项、状态、结果 |
| 政务统一认证/赣服通 | MVP 不强依赖 | 后续接入企业身份认证 |
| 政务 OA/统一身份认证 | MVP 不强依赖 | 后续接入政府端身份认证 |
| 审批系统 | MVP 先做独立辅助审核闭环 | 后续按事项对接审批状态和结果 |

### 12.2 认证授权

- MVP 自建账号体系。
- 企业端支持企业账号、经办人账号、企业统一社会信用代码绑定。
- 企业绑定采用 Agent 低风险自动审核：统一社会信用代码格式正确、企业名称一致、营业执照/OCR 信息一致时自动通过，并写入审核记录；不满足任一条件时进入平台运营人工队列。
- 政府端支持部门、角色、事项、政策库、材料字段权限。
- 管理后台支持系统管理员、政策管理员、知识库运营、质检人员分权。
- 权限采用 RBAC + ABAC：角色决定基础权限，部门、事项、材料密级决定数据边界。

### 12.3 数据同步

| 数据 | MVP 数据源 | 后续 |
| --- | --- | --- |
| 政策文件 | 后台导入 | 外部平台或部门接口同步 |
| 企业画像 | 企业手填为主，后台可导入 | 工商、税务、人社等权威数据源 |
| 申报状态 | 本系统维护 | 审批系统同步 |
| 材料文件 | 企业本次申报上传 | MVP 不建设长期材料库 |

## 13. 前端体验与视觉设计

前端必须达到企业级/政务级产品质感，不接受 demo 式页面。设计效果可参考 Pencil MCP 生成或维护的设计稿、组件层级、页面截图和布局检查结果。

| 要求 | 说明 |
| --- | --- |
| 专业可信 | 风格稳重、清晰、克制，符合政务和企业服务场景 |
| 高效办事 | 企业 3 步内看到“能申报什么、还差什么、下一步做什么” |
| 手机端完整申报 | MVP 支持手机端完整申报、材料上传和多政策填报 |
| 数据可读 | 政策匹配、资格预判、材料状态、审核进度适合快速扫描 |
| 状态明确 | 待补充、已通过、不符合、需人工复核等状态清晰区分 |
| 可解释 | AI 结论可展开查看政策依据、材料依据、风险说明 |
| 验收 | 关键页面需提供 Pencil MCP 或等价工具的截图/布局检查结果 |

## 14. 技术方案

本方案参考 `memory/agent-orchestration-standards.md`、`memory/model-selection-standards.md`、`docs/模型选型规格书.md`、`docs/Agent工具定义文档.md`、`docs/Agent编排详细设计.md`，采用“图编排 + 分层 RAG + 规则优先 + 模型分级 + 工具调用 + 人工复核”。

| 模块 | 建议 | 说明 |
| --- | --- | --- |
| 前端 | Next.js / React + Ant Design Pro + Formily | 支持桌面端和 H5 响应式手机端完整申报 |
| 设计参考 | Pencil MCP | 用于设计稿、组件层级、截图和布局验收，不作为运行时依赖 |
| RAG 引擎 | RAGFlow + 向量数据库 + 结构化规则库 | 支持政策文档解析、引用、语义检索和规则校验 |
| Agent 编排 | LangGraph 类图编排/状态机框架 + Postgres Checkpoint | 支持状态、分支、人工节点、失败恢复、trace；关键业务结果同步写业务表 |
| OCR | PaddleOCR 或政务内网既有 OCR | 中文政务材料需样本验证和人工复核 |
| 权限 | Casbin | 支持 RBAC/ABAC，适合部门、事项、材料权限 |
| 文件存储 | MinIO | 私有化对象存储，需加密、权限和生命周期策略 |
| 看板 | Apache Superset / Grafana | 业务看板和技术监控分开 |
| 观测 | OpenTelemetry + 审计表 | 模型调用、工具调用、人工动作全链路留痕 |

### 14.0 模型部署与接口

模型采用混合部署和 OpenAI-compatible 接口抽象。公开政策问答、Query 改写、意图识别可使用低成本兼容模型；企业申报材料、财务、用工、资格预判、审批辅助等敏感任务走私有化或专有云模型。候选模型包括 DeepSeek、Qwen、BGE Embedding、BGE Reranker、Whisper/PaddleSpeech 及其他 OpenAI-compatible 服务，均需通过 POC 后定版，不视为已部署事实。

### 14.1 Agent 分工

| Agent | 输入 | 输出 | 人工兜底条件 |
| --- | --- | --- | --- |
| 企业诉求 Agent | 用户问题、画像、历史对话 | 意图、缺失字段、追问问题 | 意图不明、缺关键字段 |
| 检索调度 Agent | 意图、画像、权限 | 检索策略、候选政策 | 无召回、低相似度、权限冲突 |
| 政策分析 Agent | 候选政策、画像、条件 | 规则比对、资格预判 | 条件冲突、无法结构化 |
| 申报辅助 Agent | 政策、材料、画像 | 预填表单、缺失材料 | 材料缺失、OCR 异常 |
| 审核 Agent | 申报、材料、政策条件 | 预审建议、风险点、意见草稿 | 自由裁量、证据不足、低置信度 |

Agent 工具调用详见 `docs/Agent工具定义文档.md`。所有工具必须定义函数签名、JSON Schema、错误码、超时、权限、幂等性和重试策略。

### 14.1.1 编排状态与持久化

LangGraph State 由 `session`、`user_context`、`enterprise_profile_snapshot`、`intent`、`retrieval_result`、`policy_conditions`、`eligibility_result`、`materials`、`application`、`review`、`fallback`、`audit_trace` 组成。State 使用 Postgres Checkpoint 保存恢复点，Application、ReviewRecord、AuditLog 等关键业务结果同步写业务表，业务表作为事实源。

### 14.2 POC 要求

正式开发前需用 2-3 周完成端到端 POC：

- LangGraph 持久化状态能否支撑持续数周的申报流程。
- Postgres Checkpoint + 业务表双写是否能支撑失败恢复、审计追踪和长流程恢复。
- RAGFlow 是否支持内网部署、政策版本和权限过滤。
- PaddleOCR 在真实中文政务材料上的字段识别准确率。
- 候选模型在资格预判、条件拆解、审批草稿中的延迟、准确率、结构化输出成功率和成本。
- RAG + 规则 + 模型 + 人工复核的完整 trace 是否可审计。

## 15. 性能、安全与合规

### 15.1 性能指标

| 场景 | MVP 目标 |
| --- | --- |
| 政策问答整体响应 | P95 <= 5s，缓存命中简单问答 P95 <= 3s |
| 检索召回 | P95 <= 1.5s |
| 资格预判 | P95 <= 8s，复杂多政策可异步 |
| OCR 单页处理 | P95 <= 10s，超过转异步任务 |
| 页面首屏 | 内网环境 LCP <= 2.5s |
| 并发 | MVP 支持 100 家企业试点使用、50 名政府人员并发审核 |

### 15.2 数据安全分级

| 等级 | 数据 | 处理要求 |
| --- | --- | --- |
| L1 公开 | 已公开政策文本、公开办事指南 | 可公开展示，保留来源和版本 |
| L2 内部 | 政策结构化规则、审核流配置、运营指标 | 部门授权访问，禁止外发 |
| L3 企业敏感 | 营业执照、财报、纳税、用工、合同、联系方式 | 加密存储，字段脱敏，最小权限 |
| L4 审批敏感 | 审核意见、风险标记、人工复核记录、模型 trace | 严格授权、全量审计、禁止出域 |

脱敏要求：手机号、身份证号、统一社会信用代码、银行账号、纳税金额、合同金额等在日志、看板、导出中按权限脱敏。

## 16. 异常与边界

| 场景 | 处理规则 |
| --- | --- |
| 无匹配政策 | 展示原因、建议补充画像字段、提供人工咨询入口 |
| 政策废止 | 未提交禁止申报；已提交转人工复核 |
| 企业画像变更 | 新申报使用新画像；历史申报保留提交快照 |
| 多人并发编辑 | 草稿保存提示冲突；提交和审核采用乐观锁 |
| 材料过期 | 标记不可用，提示重新上传 |
| 手机端上传失败 | 保留草稿，支持重新上传和桌面端继续办理 |
| AI 无引用回答 | 禁止展示为正式答案，转人工或提示无法确认 |
| 审批超时 | 看板预警，按部门 SLA 升级提醒 |

| 类型 | 验收标准 |

| 安全 | 敏感字段脱敏、加密、日志留痕通过检查 |

## 20. 风险与依赖

| 风险/依赖 | 影响 | 应对 |
| --- | --- | --- |
| 手机端完整申报复杂 | MVP 工作量增加 | 优先覆盖高频政策表单，复杂事项允许异步处理 |
| 外部平台暂不对接 | 状态与事项无法自动同步 | 本期独立闭环，后续扩展接口 |
| 政策结构化质量不足 | 预判不准 | 人工审核发布，先覆盖高频政策 |
| OCR 准确率不足 | 材料核验误判 | 低置信度人工确认，样本持续优化 |
| 模型延迟高 | 企业体验差 | 模型路由、缓存、异步任务、降级检索 |
| 权限配置复杂 | 越权风险 | RBAC + ABAC，默认最小权限 |
