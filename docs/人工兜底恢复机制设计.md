# 人工兜底恢复机制设计

> 目标：明确 `human_fallback_node` 如何暂停、等待人工处理、恢复 LangGraph State，并同步业务表。

## 1. 触发场景

| 场景 | 触发 |
| --- | --- |
| 检索低置信度 | Top1 < 0.75 |
| OCR 低置信度 | 字段置信度 < 0.85 |
| 规则证据缺失 | 必填字段或材料缺失 |
| 政策冲突 | 条件互斥或语义不清 |
| 模型输出异常 | JSON 格式错误重试后仍失败 |

## 2. 暂停流程

```text
Agent节点发现低置信度
  -> 写 fallback.required=true
  -> 创建 FallbackTask
  -> 写 AuditLog
  -> LangGraph interrupt
  -> Postgres Checkpoint 保存 State
```

## 3. 人工处理

平台运营在后台处理 `FallbackTask`：

| 字段 | 说明 |
| --- | --- |
| task_id | 兜底任务 ID |
| run_id | LangGraph 运行 ID |
| reason | 触发原因 |
| context_summary | 上下文摘要 |
| resolution_type | answer、field_patch、material_confirm、transfer_department、close |
| resolution_payload | 人工处理结果 |
| resolver_id | 处理人 |

普通任务 1 个工作日内处理。

## 4. Resume API

```http
POST /api/v1/agent-runs/{run_id}/resume
```

Request:

```json
{
  "task_id": "task_001",
  "resolution_type": "field_patch",
  "resolution_payload": {
    "confirmed_fields": {
      "tax_amount": 120000
    },
    "comment": "已电话确认企业纳税额"
  },
  "resolver_id": "user_001"
}
```

Response:

```json
{
  "run_id": "run_001",
  "resumed": true,
  "next_node": "policy_analysis_node"
}
```

## 5. State 恢复

1. 根据 `run_id` 从 Postgres Checkpoint 读取最近状态。
2. 校验 `task_id` 是否属于该 run。
3. 将人工处理结果写入 `fallback.resolution_payload`。
4. 按 `resolution_type` patch 对应 State 字段。
5. 继续执行下一节点。
6. 写入业务表和 AuditLog。

## 6. 业务表同步

| 人工结果 | 同步表 |
| --- | --- |
| 字段确认 | Application / EnterpriseProfileSnapshot 补充记录 |
| 材料确认 | Material / OCRResult |
| 审核判断 | ReviewRecord |
| 知识修正 | 标注回流队列 |
| 关闭任务 | FallbackTask |

## 7. 失败处理

| 失败 | 处理 |
| --- | --- |
| Checkpoint 不存在 | 以业务表为事实源重建关键 State |
| State patch 冲突 | 进入人工复核，不自动覆盖 |
| resume 后工具失败 | 按分级重试策略处理 |
| 重复 resume | 幂等返回已有结果 |
