### B. RAG 生产链路还没完全打通

你有 RAG sidecar 代码，但生产级还依赖外部环境。

* 只读 DB 用户写业务表失败

影响：营业执照 OCR 基础可用，但通用材料 OCR 和生产部署链路还没完整打通。

### 第三层：Agent 生产验收

目标：证明多 Agent 运行时能安全稳定地跑。

* API Web 和 Worker 分离
* API：

```bash
AGENT_RUN_ASYNC_ENABLED=true
AGENT_RUN_WORKER_AUTOSTART=false
```

* Worker：

```bash
AGENT_RUN_WORKER_ID=<唯一 worker id>
```

验收用例：

1. 创建 Agent Run，API 立即返回 `queued`
2. Worker 消费 job 后 run 进入 `completed` / `interrupted` / `failed`
3. start job 抛错后可 retry
4. stale job 不重放副作用
5. Worker A 丢 lease 后不能写：
   * step
   * tool call
   * checkpoint
   * fallback
   * LLM call
6. async resume 能恢复
7. resume 幂等
8. 不同 actor 下 global quota 不突破
9. daily budget overrun 能触发 alert
10. primary model circuit open 后 fallback model 生效
11. review draft 不自动审批，必须人工确认

可以用：

```bash
npm exec vitest -- run --maxWorkers=1 test/batch22.agent-runtime-controls.test.ts
npm run llm:agent-core-eval
```

## 六、最终一句话

**你的项目现在“后端核心能力 + 多 Agent runtime”已经很接近生产级，Agent 部分本地代码门禁基本打通；但整个项目还没有完全打通。**

目前最准确的状态是：

> **后端本地可提交、Agent runtime 可进入预发试点；但真实企业生产级上线还缺 RAG/OCR/对象存储/Worker 部署/前端/部分业务接口/预发运维验收。**

如果你下一步要推进，我建议先做一个  **“预发环境打通 Batch”** ，不要继续堆 Agent 功能。这个 batch 的目标就是把 DB、RAG、OCR、对象存储、API/Worker 分离部署和 `ops:preprod-agent-runtime-check` 全部跑通。
