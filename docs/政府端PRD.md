# PRD：南康助企宝政府端

> 文档状态：Draft  
> 模块定位：窗口服务、辅助审核、补正处理、审批留痕  
> 目标端：政府工作人员桌面端  
> 参考总文档：`docs/大数据中心AI产品需求文档.md`

## 1. 背景

政府端承担企业咨询承接、申报材料审核、补正通知、办理进度维护等工作。当前痛点集中在重复咨询多、政策条件人工比对慢、材料核验耗时、审批依据整理成本高。政府端通过 AI 预审和结构化比对提升效率，但最终审批必须由人工确认。

## 2. 目标用户

| 用户 | 典型角色 | 核心诉求 |
| --- | --- | --- |
| 窗口人员 | 惠企政策窗口、园区服务人员 | 快速理解企业诉求并分流事项 |
| 审批人员 | 工信、商务、人社、税务等部门审核人员 | 快速完成材料核验、条件比对和意见生成 |
| 部门负责人 | 科室负责人、审批负责人 | 掌握办理效率、积压任务和异常案例 |

## 3. 已确认范围

| 决策项 | 政府端写法 |
| --- | --- |
| 外部平台 | MVP 不对接亲清赣商/惠企通和外部审批系统，先做独立辅助审核闭环 |
| 登录认证 | MVP 自建账号体系，预留政务 OA/统一身份认证 |
| 资格预判 | 规则优先，AI 只解释和辅助总结 |
| AI 审批意见 | MVP 可生成草稿，但必须人工确认后生效 |
| 多政策申报 | 企业端合并填写后，政府端按政策子项分别审核 |
| 材料复用 | 只在本次申报内复用，不读取长期企业材料库 |
| 人工兜底 | 平台运营统一处理低置信度队列，普通任务 1 个工作日内处理 |

## 4. 问题定义

1. 大量企业咨询重复，窗口人员处理压力大。
2. 政策条件复杂，人工逐项比对效率低。
3. 材料缺失、过期、字段不一致问题发现晚。
4. 审批意见草稿和补正说明需要人工反复编写。
5. AI 参与审核后，需要可解释、可追溯、可人工纠偏。

## 5. 目标与非目标

### 5.1 目标

- 建立统一审核工作台。
- 自动生成政策条件比对清单。
- 辅助识别材料缺失、异常和风险点。
- 生成审批意见草稿和补正建议，人工确认后生效。
- 支持补正通知和全流程留痕。

### 5.2 非目标

- 不让 AI 自动作出最终审批决定。
- MVP 不对接亲清赣商/惠企通或外部审批系统。
- 不绕过部门权限和既有审批责任。
- 不在证据不足时给出确定性结论。
- MVP 不处理所有部门复杂自由裁量事项。

## 6. 解决方案

政府端以“审核工作台”为核心：

1. 任务队列：按部门、政策、状态、时限展示任务。
2. 申报详情：展示企业信息、申报政策、材料和历史记录。
3. 条件比对：将政策条件与企业数据、材料字段逐项比对。
4. AI 预审：输出建议通过、需补充、不符合、需人工复核。
5. 审批意见：生成草稿，人工确认或修改后生效。
6. 补正处理：生成补正要求并推送企业端。
7. 审计留痕：记录模型、规则、人工修改和审批动作。

## 7. 用户流程

### 7.1 审核流程

1. 审批人员进入任务队列。
2. 选择待审核事项。
3. 系统加载政策条件、企业画像快照、申报材料。
4. 审核 Agent 执行条件比对和材料校验。
5. 系统展示预审建议、依据、风险点和意见草稿。
6. 审批人员确认、修改或驳回建议。
7. 系统生成审批记录并同步企业端状态。

### 7.2 多政策子项审核

1. 企业端一次提交多政策申报。
2. 后端按政策拆成多个 `ApplicationPolicyItem`。
3. 政府端按政策子项分配部门和审核人员。
4. 每个子项独立审核、补正、通过或驳回。
5. 企业端展示主单总状态和各政策子项状态。

主单/子项 UI 交互：

- 审核任务列表默认展示子项任务，支持按主单聚合查看。
- 申报详情顶部展示主单状态、企业画像快照、提交时间、总进度和风险摘要。
- 左侧或顶部展示政策子项列表，字段包括政策名称、责任部门、子项状态、预审结果、超时风险。
- 选择某个子项后，右侧展示该政策的条件比对、材料引用、AI 预审草稿和人工操作区。
- 子项可独立通过、驳回、补正或转人工；主单状态由子项状态汇总生成。
- 发起补正时可选择仅补某个子项，也可选择影响多个子项的共用材料。

### 7.3 补正流程

1. 系统识别材料缺失或字段异常。
2. 审批人员查看补正建议。
3. 审批人员编辑或确认补正文案。
4. 企业端收到补正通知。
5. 企业补正后任务回到审核队列。

### 7.4 人工复核流程

1. 系统遇到低置信度、政策冲突、材料不一致、自由裁量事项。
2. 任务标记为需人工复核。
3. 审批人员查看触发原因。
4. 人工处理结果回流知识库和标注队列。

## 9. 数据模型

| 实体 | 政府端使用字段 | 要求 |
| --- | --- | --- |
| Application | application_id、enterprise_id、status、submit_time、deadline_at、profile_snapshot_id | 展示主单状态和企业画像快照 |

## 10. 状态机

| 状态 | 政府端含义 | 可操作 |
| --- | --- | --- |
| submitted | 企业已提交 | 发起预审 |
| pre_reviewing | 系统预审中 | 查看预审结果 |
| reviewing | 人工审核中 | 通过、驳回、补正、复核 |
| need_supplement | 已发起补正 | 查看补正状态 |
| resubmitted | 企业已补正 | 重新预审或审核 |
| manual_review | 需人工复核 | 人工判断并留痕 |
| approved | 已通过 | 查看归档 |
| rejected | 不通过 | 查看原因 |
| timeout_closed | 超时关闭 | 查看归档 |

## 11. 规则与阈值

### 11.1 低置信度转人工

| 触发条件 | 初始阈值或规则 | 处理 |
| --- | --- | --- |
| 检索 Top1 相似度低 | < 0.75 | 不给确定结论，转人工复核 |
| OCR 字段置信度低 | < 0.85 | 审批人员人工确认 |
| 规则证据缺失 | 必填条件无证据 | 标记待补充 |
| AI 与规则冲突 | 任意冲突 | 以规则为准，转人工 |
| 自由裁量事项 | 无结构化硬规则 | 人工复核 |

阈值为初始值，需 POC 校准。

### 11.2 审批意见草稿

- AI 可以生成审批意见草稿。
- 草稿默认不得自动提交。
- 审批人员必须确认或修改后才能生效。
- 系统记录草稿原文、人工修改内容、确认人和确认时间。

## 12. 前端体验要求

- 政府端优先桌面办公效率，支持高密度信息扫描。
- 审核工作台以表格、筛选、批量操作、详情抽屉/分屏为主。
- 条件比对结果必须可快速判断：满足、不满足、缺材料、需人工复核。
- AI 预审建议不能遮蔽人工判断入口。
- 页面需提供政策依据和材料依据展开能力。
- 关键页面设计稿、截图和布局检查可参考 Pencil MCP。

## 14. 数据与指标

| 指标 | 说明 |
| --- | --- |
| 平均审核耗时 | 单个事项从进入审核到完成的平均时间 |
| AI 预审采纳率 | 审批人员采纳 AI 建议的比例 |
| 审批意见草稿使用率 | AI 草稿被确认或修改后使用的比例 |
| 补正率 | 需企业补正的事项比例 |
| 人工复核率 | 进入人工复核的事项比例 |
| 超时任务率 | 超过承诺时限的任务比例 |

## 15. 验收标准

| 类型 | 标准 |
| --- | --- |
| 条件比对 | 每个结论能追溯到政策条件、企业字段或材料字段 |
| AI 草稿 | 未经人工确认不得生效，确认和修改全量留痕 |
| 低置信度 | 触发阈值后必须进入人工复核 |
| 多政策子项 | 一个主单下多个政策子项可独立审核和展示状态 |
| 权限 | 不同部门只能查看授权事项和材料字段 |
| 前端 | 关键页面通过 Pencil MCP 或等价截图/布局检查 |

## 16. 风险与依赖

| 风险/依赖 | 影响 | 应对 |
| --- | --- | --- |
| AI 建议被误用为审批结论 | 合规风险 | 明确人工确认，页面标注“辅助建议” |
| 权限配置不当 | 数据越权风险 | 使用部门、角色、事项多级权限校验 |
| 审批人员不信任 AI | 采纳率低 | 展示依据、风险点和可修改草稿 |
| 材料字段识别不准 | 影响预审质量 | OCR 低置信度转人工确认 |
| 本期不接外部审批系统 | 状态需要本系统维护 | MVP 独立闭环，后续再评估接口 |
